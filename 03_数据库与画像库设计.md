# 数据库与画像库设计

## 1. 这份文档解决什么问题

这份文档专门回答下面几个关键问题：

- 说话人注册信息以后怎么存
- 原始音频和 embedding 怎么对应
- “画像库”到底是什么
- benchmark 结果以后怎么沉淀

白话理解：

- 以前更像“登记几段音频”
- 后面要升级成“给每个人建立完整档案”

## 2. 当前现状

当前 `voice-benchmark` 里已经有一个基础版注册库：

- 数据库存 `speakers`
- 数据库存 `speaker_audio_files`
- 本地目录存原始音频文件

这个方向本身是对的。

但它还不够支撑后面的自研框架，因为现在主要还是：

- 管理“文件”

而不是：

- 管理“说话人画像”

## 3. 目标状态

后面的数据库设计，要从“文件登记簿”升级成“画像系统”。

目标状态里，数据库至少要能表达下面几层信息：

1. 谁是这个人
2. 这个人有哪些原始注册音频
3. 每条音频对应什么 embedding
4. 每条样本质量怎么样
5. 这个人的中心画像是什么
6. 这个画像由哪一版规则生成
7. 某次 benchmark 跑的是哪版画像和哪版策略

## 4. 设计原则

数据库设计必须遵守下面几个原则：

- 原始音频和计算结果分开存
- 原子样本和聚合画像分开存
- 当前有效版本和历史版本都能追溯
- benchmark 结果能回查到当时所用的模型、阈值、策略
- 尽量不破坏现有库，而是在其上平滑扩展

## 5. 核心数据层次

后面建议把数据分成 5 层。

### 5.1 说话人层

这一层描述“这个人是谁”。

例如：

- `speaker_id`
- `speaker_name`
- 状态
- 创建时间
- 更新时间

### 5.2 原始样本层

这一层描述“这个人注册时交了哪些原始音频”。

例如：

- 原始文件名
- 存储路径
- 文件大小
- 音频时长
- 采样率
- 声道数
- 来源

### 5.3 embedding 样本层

这一层描述“每条样本在某个 backend 下提出来的向量结果”。

注意：

- 同一条原始音频
- 用不同 backend
- 可能得到不同 embedding

所以 embedding 不能只跟 speaker 绑死，还要跟 backend 和版本绑住。

### 5.4 画像层

这一层描述“这个人最终的聚合档案”。

也就是说：

- 不是一条音频
- 不是一条 embedding
- 而是多个 embedding 按某种规则聚合后的结果

### 5.5 评测层

这一层描述：

- 某次 benchmark 是怎么跑的
- 跑了哪些样本
- 用了哪个 backend
- 用了什么阈值
- 得到了什么结果

## 6. 推荐数据表

后面建议逐步演进为下面几张表。

### 6.1 `speakers`

作用：

- 存说话人主档案

建议字段：

- `id`
- `speaker_id`
- `speaker_name`
- `status`
- `created_at`
- `updated_at`

说明：

- 这张表可以基本沿用当前设计

### 6.2 `speaker_audio_files`

作用：

- 存原始注册音频元数据

建议字段：

- `id`
- `speaker_db_id`
- `original_filename`
- `stored_filename`
- `storage_path`
- `file_size`
- `duration_sec`
- `sample_rate`
- `channels`
- `source_type`
- `created_at`

建议新增字段的原因：

- 后面质量评估和筛样本时，会用到时长、采样率、来源

### 6.3 `speaker_embeddings`

作用：

- 存每条原始样本在某个 backend 下提出来的 embedding 结果

建议字段：

- `id`
- `speaker_db_id`
- `audio_file_id`
- `backend_name`
- `backend_version`
- `feature_version`
- `embedding_dim`
- `embedding_path`
- `norm_value`
- `duration_sec`
- `quality_score`
- `is_active`
- `created_at`

说明：

- `embedding_path` 不建议直接把大向量塞进 PostgreSQL
- 更建议落成文件，再在数据库里存路径和元数据

白话理解：

- 数据库像目录索引
- embedding 文件像真正的档案袋

### 6.4 `speaker_profiles`

作用：

- 存聚合后的说话人画像

建议字段：

- `id`
- `speaker_db_id`
- `profile_name`
- `backend_name`
- `backend_version`
- `feature_version`
- `aggregation_strategy`
- `profile_vector_path`
- `profile_dim`
- `sample_count`
- `avg_quality_score`
- `is_active`
- `created_at`

说明：

- 同一个人未来可以有多个 profile
- 例如：
  - `default`
  - `high_quality_only`
  - `multi_center`

### 6.5 `speaker_profile_members`

作用：

- 描述某个画像由哪些 embedding 组成

建议字段：

- `id`
- `profile_id`
- `embedding_id`
- `weight_value`
- `member_role`

作用很关键：

- 以后回头查“这个画像怎么来的”，就靠这张表

### 6.6 `evaluation_runs`

作用：

- 记录某次 benchmark 或专项评测的整体信息

建议字段：

- `id`
- `run_name`
- `run_type`
- `backend_name`
- `backend_version`
- `feature_version`
- `scoring_strategy`
- `threshold_value`
- `dataset_name`
- `dataset_version`
- `report_path`
- `table_path`
- `summary_json`
- `created_at`

### 6.7 `evaluation_items`

作用：

- 记录某次评测里每一条测试样本的结果

建议字段：

- `id`
- `evaluation_run_id`
- `clip_name`
- `truth_label`
- `expected_label`
- `best_score`
- `threshold_value`
- `decision`
- `final_label`
- `is_correct`
- `latency_ms`
- `score_breakdown_json`

### 6.8 `hard_cases`

作用：

- 记录误判难例

建议字段：

- `id`
- `clip_name`
- `speaker_label`
- `expected_label`
- `predicted_label`
- `backend_name`
- `error_type`
- `note`
- `status`
- `created_at`

作用：

- 后面做回归测试时很值钱

## 7. 画像库到底是什么

“画像库”这个词，必须说清楚。

这里说的画像库，不是玄学概念，而是：

- 某个说话人的一组高质量表征集合

它至少应该包含三样东西：

1. 原始注册样本集合
2. 这些样本计算出来的 embedding 集合
3. 按某种规则聚合后的最终画像

所以以后“注册一个人”的真正流程应该是：

1. 保存原始音频
2. 计算 embedding
3. 打质量分
4. 生成 profile
5. 标记当前 active profile

## 8. 为什么 embedding 不建议直接塞数据库

我建议 embedding 向量文件落盘，数据库只存索引信息。

原因：

- 向量数据体积不小
- 后面会有多 backend、多版本
- 直接塞表里会让数据库越来越重
- 文件更适合后续批量重建和迁移

建议做法：

- embedding 文件统一落在工作区某个固定目录
- 数据库只存：
  - 路径
  - 维度
  - backend
  - 版本
  - 质量分

## 9. 版本管理建议

后面一定要加版本概念，不然后面越跑越乱。

至少要有下面几类版本：

- `backend_version`
- `feature_version`
- `profile_version`
- `dataset_version`
- `benchmark_version`

为什么要这么做：

- 同一个人今天的画像和下周的画像可能不同
- 同一个 backend 换了前处理，分数也会变
- 如果没有版本号，后面很难知道“这次结果到底和上次差在哪”

## 10. 与现有库的兼容方式

不建议一上来推翻现有表。

建议分两步升级。

### 10.1 第一步

保留现有：

- `speakers`
- `speaker_audio_files`

新增：

- `speaker_embeddings`
- `speaker_profiles`
- `speaker_profile_members`

这样可以最小代价跑通第一版画像库。

### 10.2 第二步

等 benchmark 和画像逻辑稳定后，再新增：

- `evaluation_runs`
- `evaluation_items`
- `hard_cases`

这样可以避免一开始把系统改得太重。

## 11. 第一阶段最小可用设计

如果只考虑 V1，最小可用表设计其实只需要这 5 张：

- `speakers`
- `speaker_audio_files`
- `speaker_embeddings`
- `speaker_profiles`
- `speaker_profile_members`

靠这 5 张表，就已经能支撑：

- 多样本注册
- embedding 缓存
- 中心画像
- `max / center` 打分
- 画像重建

## 12. 典型业务流程

### 12.1 注册流程

建议后面统一为：

1. 用户上传原始注册音频
2. 写入 `speaker_audio_files`
3. 调用指定 backend 计算 embedding
4. 写入 `speaker_embeddings`
5. 根据规则聚合生成画像
6. 写入 `speaker_profiles`
7. 建立 `speaker_profile_members`
8. 标记当前 active profile

### 12.2 识别流程

建议后面统一为：

1. 输入测试片段
2. 计算测试 embedding
3. 加载当前 active profiles
4. 运行打分策略
5. 输出 accept / reject / unknown

### 12.3 benchmark 流程

建议后面统一为：

1. 选择数据集版本
2. 选择 backend 和策略
3. 自动跑批
4. 写入 `evaluation_runs`
5. 写入 `evaluation_items`
6. 生成中文报告和总表

## 13. 文件落盘建议

后面建议在工作区中留出专门的数据目录，例如：

```text
data/
  enrollments/
  embeddings/
  profiles/
  evaluations/
```

说明：

- `enrollments/` 放原始注册音频
- `embeddings/` 放向量文件
- `profiles/` 放画像向量文件
- `evaluations/` 放评测产物

## 14. 当前决定

从现在开始，数据库设计正式按下面的方向走：

- 保留现有注册主表
- 新增 embedding 层
- 新增 profile 层
- 后续新增 evaluation 层

也就是说：

- 后面不再只做“音频文件登记”
- 而是正式升级到“说话人画像系统”
